# Satori knot algorithm

[https://chat.z.ai/c/85c5af83-50f4-4a91-96ef-753c27f6902e](https://chat.z.ai/c/85c5af83-50f4-4a91-96ef-753c27f6902e)

### **Overview**

A Satori drawing is generated by scattering a number of focal points (called **"knots"**) across a 2D canvas. The grayscale value (0–255) of each pixel is calculated based on its distance and angle to these knots. The drawing can blend two styles simultaneously by evaluating them in a checkerboard pattern across the pixels.

---

### **1. Initialization (Generating Knots)**

When creating a drawing, you must generate an array of `num_knots` knots. Each knot has a position and a set of random attributes that control the 5 different styles.

**Positioning:** The knots are randomly placed within a slightly zoomed-in area of the canvas.

* `zoom = 1.1`  
* `origin_x = 0.5 * (zoom - 1.0) * width`  
* `origin_y = 0.5 * (zoom - 1.0) * height`  
* For each knot `i`:  
  * `knot[i].x = zoom * width * random_uniform(0, 1) - origin_x`  
  * `knot[i].y = zoom * height * random_uniform(0, 1) - origin_y`

**Random Attributes:** For each knot `i`:

* `flowsign`, `spinsign`, `leafsign`, `rayssign`, `wavesign` = Randomly choose either `1.0` or `-1.0` (50/50 chance).  
* `nspokes` = Random integer between `1` and `7` inclusive.  
* `sectors` = `nspokes / (2 * π)`  
* `frequency` = `6 * random_uniform(0, 1) + 3` *(Range: 3.0 to 9.0)*  
* `amplitude` = 50% chance of `0`, otherwise `8 * frequency / (nspokes²)`  
* `decay` = `20 + 30 * random_uniform(0, 1)` *(Range: 20.0 to 50.0)*

**Global Style Parameters:**

* `leafdiscrete` = `1 + 3 * random_integer(0, 2)` *(Results in 1, 4, or 7)*  
* `raysdiscrete` = `1 + 3 * random_integer(0, 2)` *(Results in 1, 4, or 7)*

---

### **2. Pixel Evaluation**

To determine the value of a pixel at coordinates `(x, y)`:

1. Determine which style to use based on a checkerboard pattern:  
   * If `(x + y) % 2 == 0`, use `style1`.  
   * Otherwise, use `style2`.  
2. Pass the pixel coordinates to the math function for the chosen style.  
3. The final pixel value must wrap into an 8-bit unsigned integer (0–255). Because the C code relies on casting negative numbers to `unsigned char`, a true cross-language equivalent is: `final_value = (computed_value % 256 + 256) % 256`.

---

### **3. The Style Algorithms**

For all styles, let `dx = x - knot.x` and `dy = y - knot.y`.

#### **Flow**

Based on the logarithm of the distance to the knots.

```python  
val = 0.0  
for each knot:  
    val += knot.flowsign * log(dx² + dy²)

val *= int(100 / num_knots)  # Note: Integer division  
return int(val) % 256  
```

#### **Wave**

Similar to flow, but passes the logarithmic distance through a sine wave to create reflecting color bands.

```python  
val = 0.0  
for each knot:  
    val += knot.wavesign * sin(1.5 * log(dx² + dy²))

val *= int(100 / num_knots)  # Note: Integer division  
return int(val) % 256  
```

#### **Spin**

Based on the angle to the knot. Creates a radially slicing effect, with optional waviness.

```python  
val = 0.0  
for each knot:  
    r = sqrt(dx² + dy²)  
      
    if r == 0:  
        a = 0.0  
    else:  
        a = atan2(dy, dx)  
      
    # Apply exponential-decay waviness  
    a += knot.amplitude * knot.sectors * sin(r / knot.frequency) * exp(-r / knot.decay)  
      
    # The C code does: knot.sectors * fmod(a, 1.0 / knot.sectors)  
    # This is mathematically identical to fmod(a * knot.sectors, 1.0)  
    a = fmod(a * knot.sectors, 1.0)   
      
    val += knot.spinsign * a

return int(256 * val) % 256  
```

#### **Leaf**

Based on the ratio of the smaller axis offset to the larger axis offset, creating corner-cutting diamond/leaf shapes. The `leafdiscrete` variable forces color quantization.

```python  
val = 0  
for each knot:  
    big = max(abs(dx), abs(dy))  
    small = min(abs(dx), abs(dy))  
      
    if big == 0:  
        a = 0.0  
    else:  
        a = knot.leafsign * 75 * (small / big)²  
          
    # Quantize the value  
    val += (int(a) / leafdiscrete) * leafdiscrete  # Note: Integer division

return int(val) % 256  
```

#### **Rays**

*(Note: The author's comment claims this varies quadratically compared to Leaf, but the actual C source code is mathematically identical to Leaf except it uses the `rays` parameters instead of `leaf` parameters.)*

```python  
val = 0  
for each knot:  
    big = max(abs(dx), abs(dy))  
    small = min(abs(dx), abs(dy))  
      
    if big == 0:  
        a = 0.0  
    else:  
        a = knot.rayssign * 75 * (small / big)²  
          
    # Quantize the value  
    val += (int(a) / raysdiscrete) * raysdiscrete  # Note: Integer division

return int(val) % 256  
```

### **⚠️ Critical Implementation Details for Reimplementation**

If you are porting this to another language, you must account for these C-specific behaviors, or your output will look completely wrong:

1. **Integer Division:** In `flow` and `wave`, `100 / num_knots` is integer division (e.g., `100 / 3 = 33`). In `leaf` and `rays`, the quantization math `(int(a) / discrete) * discrete` also relies on integer division.  
2. **Truncation towards Zero:** C's integer division and `(int)` casting truncate towards zero. For example, `-5 / 2` is `-2` in C, whereas Python's floor division (`//`) would yield `-3`. Ensure your target language matches this behavior.  
3. **Floating-point Modulo:** C's `fmod(a, b)` returns a result with the same sign as `a`. In languages like Python or JavaScript, the `%` operator might use floor division modulo, which flips the sign for negative numbers. If your language uses floor modulo, use a function like `math.fmod()` to replicate C exactly.  
4. **Log of Zero:** In the `flow` and `wave` algorithms, if a pixel lies exactly on a knot, `dx` and `dy` are `0`. `log(0)` yields `-Infinity`. In C, this results in undefined behavior that usually silently produces an artifact. In other languages (like Python), this will throw a runtime exception. You should add a guard like `if (dx==0 and dy==0): continue` to prevent crashes.

# Palette Data

Here is the explanation of how the 0–255 grayscale values calculated by the drawing algorithms are transformed into full RGB colors using the palette system.

### **Overview: The Color Look-Up Table (LUT)**

Instead of calculating an RGB color directly for every pixel, the system uses a **two-step process**:

1. The drawing algorithm calculates a grayscale "index" from `0` to `255`.  
2. That index is fed into a pre-computed **Color Table** of 256 RGB entries. The RGB value at that index becomes the pixel's final color.

The palette files dictate how this 256-entry Color Table is built. The final table smoothly blends between a set of "key colors", creating a gradient.

---

### **Step 1: Reading the Palette Data**

A palette file (like `palettes.txt`) defines a list of palettes. Each palette provides a small number of "key colors" (usually 4 to 10).

Format: `Name NumberOfColors Color1 Color2 ...` Example: `Cold 4 0x33ccff 0x0099ff 0x0033cc 0x0033ff`

The file is read and parsed into an array of RGB structs. A hex color like `0x33ccff` is split using bitmasks:

* **R** = `(color & 0xff0000) >> 16` *(0x33)*  
* **G** = `(color & 0x00ff00) >> 8` *(0xcc)*  
* **B** = `(color & 0x0000ff)` *(0xff)*

---

### **Step 2: Generating the 256-Entry Color Table**

The core algorithm is in `get_colortable()`. It takes the small list of key colors and expands it into a smooth, continuous 256-color gradient.

**A. Determine the number of "Bands" (`nsteps`)** Depending on the boolean flags `randomize` and `stripes`, the system decides how many color segments (bands) the gradient will have:

* If `randomize` is false: `nsteps` = the number of colors in the palette.  
* If `randomize` is true: `nsteps` = a random number between 5 and 10 (or 3 to 5 if `stripes` is true).  
* If `stripes` is true: `nsteps` is doubled (to make room for black bands in between the colors).

**B. Select the Key Colors for the Bands** An array `colors[nsteps]` is filled:

* If `stripes` is enabled, every odd index is filled with pure black `{0,0,0}`.  
* If `randomize` is enabled, the even indices (or all indices, if no stripes) are filled with *randomly chosen* colors from the palette. (Note: the same palette color can be picked multiple times).  
* If neither, the array is filled sequentially with the palette's colors.

**C. Blend the Colors into 256 Slots** The 256 slots are evenly divided among the `nsteps` bands. For each band `ii`, the start color is `colors[ii]` and the end color is `colors[(ii + 1) % nsteps]` (the modulo makes the gradient loop seamlessly back to the beginning).

For each index `idx` within a band's slice of the 256 slots, a blending ratio `t` is calculated:

```text  
t = (nsteps / 256.0) * (idx - (ii * 256 / nsteps))  
```

`t` scales linearly from `0.0` (at the start of the band) to `1.0` (at the end of the band).

The final RGB color is a linear interpolation (lerp) between the start and end colors, using `t`:

```text  
mix.r = round(start.r * (1.0 - t) + end.r * t)  
mix.g = round(start.g * (1.0 - t) + end.g * t)  
mix.b = round(start.b * (1.0 - t) + end.b * t)  
```

*(The C code adds 0.5 and casts to int to achieve rounding).*

This is stored in `ctable[idx]`.

### **Step 3: The Animation Trick (Duplication)**

If you look closely at `get_colortable`, the resulting `ctable` must be allocated as `256 * 3 * 2` bytes (512 entries, not 256).

After filling indices `0` to `255`, the algorithm copies those exact values into indices `256` to `511`:

```c  
ctable[3*(idx+256) + 0] = mixcolor.r;  
ctable[3*(idx+256) + 1] = mixcolor.g;  
ctable[3*(idx+256) + 2] = mixcolor.b;  
```

**Why?** This is a highly efficient trick for **color cycling** (animation). By shifting the starting offset of the look-up table by 1 every frame, the colors appear to flow smoothly across the shapes. Because the table is duplicated, if a pixel's grayscale value is `250` and the animation frame offset is `10`, the index is `250 + 10 = 260`. Instead of writing logic to wrap around modulo 256 (`(250+10)%256 = 4`), the program can just read index `260` directly from the extended array, saving CPU cycles—a common optimization for older machines/screensavers.

### **Summary: Rendering a Pixel**

To render a given pixel `(x, y)` on frame `F`:

1. Calculate the base grayscale index `V` (from `0` to `255`) using the drawing algorithms.  
2. Calculate the final color table index: `i = V + (F % 256)`.  
3. Fetch the RGB values:  
   * `R = ctable[3 * i]`  
   * `G = ctable[3 * i + 1]`  
   * `B = ctable[3 * i + 2]`

# Behind the patterns

### **1. The Mathematical Root: Complex Logarithms and Electrostatics**

If you treat the 2D coordinates `(x, y)` as the real and imaginary parts of a complex number z, the various "styles" in the code map directly to complex analysis and 2D physics:

* **Flow:** The formula `log(dx² + dy²)` is exactly the 2D electrostatic potential of a point charge (or the real part of the complex logarithm,  
   ℜ(ln(z))  
  ). The `flowsign` (+1/-1) acts as the "charge" of the knot. The resulting images are topographic contour lines of electrostatic fields, which is why they look like magnetic fields or fluid flow around sources and sinks.  
* **Wave:** The formula `sin(1.5 * log(...))` creates wave interference. Because the frequency of the sine wave is tied to the *logarithm* of the distance, the resulting waves are "self-similar"—the spacing between the rings scales perfectly as you zoom in or out. This creates moiré-like interference patterns when multiple knots interact.  
* **Spin:** The formula uses `atan2(dy, dx)`, which is the complex argument  
   arg(z)  
   (or the imaginary part of the complex logarithm,  
   ℑ(ln(z))  
  ). By multiplying the angle by `sectors`, it slices the plane into pie pieces. Adding the radial sine wave (`sin(r/frequency)`) twists these slices into spirals, mimicking vortices in fluid dynamics.  
* **Leaf / Rays:** The ratio `min(|dx|, |dy|) / max(|dx|, |dy|)` is a clever way to map hyperbolic coordinates. It effectively measures angles using the  
   L1​  
   (Manhattan) and  
   L∞​  
   (Chebyshev) norms rather than standard Euclidean angles, which is why the resulting shapes are square/diamond-shaped (like leaves or rays) rather than circular.

### **2. The Rendering Technique: Palette Cycling (Color Shifting)**

The most recognizable retro aspect of this code is how it handles animation.

As noted in the C comments: *"the animation is done by changing only the color table."* This is a famous optimization called **Palette Cycling** (or Color Shifting), widely used in the DOS demoscene, Amiga trackers, and early operating system screensavers (like the classic *Electric Sheeps* or *Moiré* screensavers).

Instead of recalculating the math for every pixel on every frame (which would be incredibly slow on 90s hardware), the algorithm calculates the static "height map" once, mapping it to indexes `0–255`. To animate, the program simply adds an offset to the index before looking up the color. Because the colors in the table blend smoothly in a loop, shifting the offset makes the colors appear to "flow" along the contour lines—making static potential fields look like flowing, animated liquid or plasma.

### **3. The Double-Buffer Palette Trick**

The duplication of the color table into 512 bytes (`ctable[3*(idx+256)]`) is a highly recognizable, old-school optimization trick specifically for palette cycling.

If you want to shift the palette by 10, the naive approach requires wrapping the index with modulo: `color = table[(index + 10) % 256]`. Modulo operations used to be very CPU-expensive. By writing the first 256 colors into the second 256 slots, the code can just do `color = table[index + 10]` without any modulo logic. If `index` is 250 and the shift is 10, it safely reads slot 260, which is an exact copy of slot 4.
